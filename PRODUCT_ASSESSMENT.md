## 1. 長所 (Strengths)

### 1.1 機能の網羅性 — バックエンド
- FastAPI モノリス `main/api_server.py` に 225 のエンドポイント宣言。マーケットプレイスのコア(出品・検索・カート・チェックアウト・レビュー)に加え、以下がすべて実装済み:
  - プロモコード / バンドル / ギフトカード / チップ / 返金申請
  - 紹介プログラム(登録時適用 → 初回購入でボーナス付与、監査 #23)
  - ライセンスキー発行・照合、出品バージョン履歴、価格履歴
  - コミッション(制作依頼)、クリエイター申請、クリエイターダッシュボード / 収益サマリー
  - フォロー / フィード、ウィッシュリスト、保存検索、コレクション、通知(種類別ミュート)
  - VRChat 向け実用ツール(パラメータ予算計算・パフォーマンスランク推定、監査 #39〜#40)
  - 2FA、パスワードリセット(監査 #42)、メール認証、アカウント削除

### 1.2 フロントエンドの配線完了度
- React 18 + TypeScript + TanStack Query v5 で 35 ページ。バックエンド完成済み・フロント未接続だった機能を 20 ラウンドかけて全て接続済み(監査 #21〜#42)。
- ルート単位のコード分割(React.lazy)、共有クエリキャッシュによる id→名前解決、Idempotency-Key による二重決済防止(チェックアウト / バンドル / チップ / ギフトカード)。
- 型定義 `frontend/src/types/api.ts` はバックエンド `to_dict()` の実出力を読んで正確にミラーしている(推測で書いていない)。

### 1.3 検証済みであること
- バックエンド: unittest 約 1,200 件超。うち重要なのはサブプロセス型パッケージ文脈テスト(`TestVRChatToolsPackagedImport`) — 通常スイートでは隠れる import 事故を検出する。
- フロントエンド: 全ラウンドで実 uvicorn + 実ブラウザ(Playwright/Chromium)の E2E を実施し、/api/ への 400 以上の応答を全捕捉して確認。スクリーンショット付き。
- 実バグを 7 件以上発見・修正済み(下記 1.4)。「動くはず」ではなく「動作を確認した」状態。

## 2. 短所 (Weaknesses)

### 2.1 インメモリ永続化(最大の制約)
全ストアがプロセス内メモリのシングルトン。**サーバー再起動で全データが消える。** デモ・開発には十分だが、商用運用には永続化層(SQLite/PostgreSQL)が必須。移行はデータモデル全体に波及するため事業判断として保留中(`FEATURE_AUDIT.md` §3-4)。

### 2.2 実決済が存在しない
クレジットは内部通貨のみ。Stripe 等による入金、クリエイターへの出金(換金)は未実装。料金体系・手数料率・法務(資金決済法)の判断が必要なため意図的に保留(§3-1, §3-2)。

### 2.3 メール送信がない
パスワードリセットのトークンは `COCOA_EXPOSE_RESET_TOKEN=true` のとき API 応答で返す開発モードのみ。メール認証・通知メールも同様に送信基盤がない。

### 2.4 管理者 UI がない
通報(出品 / レビュー)、返金申請、クリエイター申請はバックエンド API はあるが、審査・承認を行う管理画面がない。現状は API 直叩きでしか運用できない。

### 2.5 CI がない
リポジトリに GitHub Actions 等のチェックが一切ない(status total_count 0)。かつ**このセッションの git トークンは `.github/workflows/*` と tag を push できない(403)** ため、CI 追加はリポジトリオーナーの手作業が必要。

### 2.6 フロントエンドの自動テストがない
検証は E2E スクリプト(セッション内で都度実行)のみで、リポジトリにコミットされた frontend テスト(Vitest 等)がない。回帰防御はバックエンド unittest に偏っている。

### 2.7 その他
- i18n なし(日本語ハードコード)。
- モノリス構造: `main/api_server.py` が非常に大きく、APIRouter 分割されていない。
- ミューテーションの多くが `useMutation` でなく手書き try/catch(動作は正しいがボイラープレート)。
- デッドコード: リポジトリ全体では到達不能モジュールが多い(§4-1)。`main` パッケージが実行される正典であり、ルートのフラット版は旧世代の残骸(§4-2)。

## 3. 改善案 (Improvement Proposals) — 優先度順

**P1 = 商用化のブロッカー / P2 = 品質・運用上重要 / P3 = 磨き込み**

| 優先度 | 項目 | 内容 | 判断者 |
|---|---|---|---|
| P1 | 永続化層 | SQLAlchemy + SQLite(開発)/PostgreSQL(本番)へ移行。ストアの読み書きが各 Manager クラスに閉じているため、Manager 単位で段階移行可能 | **事業判断が必要 — 提案のみ** |
| P1 | 実決済 | Stripe Checkout でクレジット購入、Stripe Connect で出金 | **事業判断が必要 — 提案のみ** |
| P1 | CI 導入 | unittest + `npm run build` + `lint` を PR ごとに実行する workflow。**オーナーが手で追加する必要あり**(トークン権限制約) | オーナー |
| P2 | メール送信抽象化 | `EmailSender` インターフェース + 開発用コンソール実装 + SMTP/SendGrid 実装。リセット / 認証 / 通知を接続 | 実装可 |
| P2 | 管理者 UI | 通報審査・返金承認・クリエイター申請審査の管理画面。権限モデルの確認が先 | 要権限設計 |
| P2 | frontend テスト | Vitest + Testing Library で主要ページのスモーク + サービス層の単体テスト。E2E スクリプトの資産をリポジトリに移す | 実装可 |
| P3 | i18n | 文言の辞書化(ja → ja/en) | 実装可 |
| P3 | 残小粒機能 | メール認証の再送 UI / Creator ページのフォロワー・フォロー中一覧 / リーダーボード・トレンドタグのウィジェット | 実装可 |
| P3 | リファクタ | 手書きミューテーションの `useMutation` 化、`api_server.py` の APIRouter 分割 | 実装可 |

### 実装禁止の境界(重要)
以下は**事業・法務・金銭の判断を伴うため、ユーザーの明示指示なしに実装してはならない**(`FEATURE_AUDIT.md` §3 に記録済み):
1. Stripe 等によるクレジット購入(実金銭の受領) — §3-1
2. クリエイター出金・換金 — §3-2
3. 金融データの削除ポリシー(アカウント削除時の取引記録の扱い) — §3-3
4. 永続化層への移行(データモデル凍結を伴う) — §3-4

## 4. 次に着手するなら

事業判断を要しない範囲で価値が高い順:
1. **メール送信抽象化(P2)** — パスワードリセットと認証が「本物」になる最短経路
2. **frontend テストのコミット(P2)** — 既存 E2E スクリプト資産の恒久化
3. **P3 小粒機能** — 各 1 ラウンド(半日相当)で完了する規模

作業手順・環境の落とし穴・検証コマンドはすべて `HANDOFF_INSTRUCTIONS.md` を参照。
