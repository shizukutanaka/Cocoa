01

前提: 製品の実像

着手前に誤解しやすい 2 点を先に固定する。

製品の実体は「アバターパラメータの取引所」

3D モデルファイルの配布基盤ではない。main/avatar_marketplace.py の MarketplaceListing dataclass にファイル/モデル URL に相当するフィールドは 存在せず（thumbnail_url のみ）、購入処理 download() が買い手に 返すのは parameters（dict）である。README.md 自身が "Preset Management: Large-scale parameter handling" と宣言しており、 これは欠陥ではなく意図されたスコープ（アセット著作権の複雑さを回避する設計） と判断する。

実行形態は単一 FastAPI モノリス

main/api_server.py（約 5,100 行・221 エンドポイント）が全 API を提供し、 main/main.py は tkinter デスクトップ GUI ランチャー。 マイクロサービス分割はコード上に存在しない。

03

不足 — 未対応

いずれも「一行修正」ではなく製品・法務レベルの判断を要するため、実装せず所見として残す。優先度順。

3-1. クレジット購入フロー（実通貨 → クレジット）が存在しない

優先度: 最高

Stripe 連携はサブスクリプション課金専用（mode == "subscription"）。 全 webhook ハンドラを確認したが、どれも marketplace_store.credit() を呼ばない。 credit() の全呼び出し元はギフトカード還元・紹介ボーナス・返金・サインアップボーナスの 4 系統のみで、実通貨からの入金経路がゼロ。サインアップボーナスを使い切ったユーザーに 追加入手手段が無く、有償マーケットプレイスが商売として成立しない。

証拠: main/billing_service.py の _WEBHOOK_HANDLERS 登録先を全確認
必要な実装: Stripe one-time payment + webhook + 冪等性処理（main/idempotency.py 再利用可）

3-2. クリエイター売上の現金化が存在しない

優先度: 高

Stripe Connect・payout・transfer に類する機構が皆無。3-1 と合わせ、クレジット経済は 入口・出口とも実通貨と断絶した閉鎖系。クリエイターは売上をプラットフォーム内でしか使えない。 資金移動業などの法規制判断を伴うため、技術実装より先にビジネス判断が必要。

証拠: grep -rin "connect\|payout\|transfer\|cashout" main/billing_service.py → 送金関連ヒットなし

3-3. アカウント削除のカスケードが存在しない

優先度: 高（法的リスク）

auth_manager.delete_user() は auth 内部の 3 インデックスから pop するのみ。 マーケットプレイス残高・取引台帳・公開リスティング・カート・ウィッシュリスト・コレクション・ コミッション・紹介コード・会員ティア・ライセンスキー・2FA 秘密鍵とバックアップコード が全て残存する。さらに管理者専用エンドポイントのみで、ユーザー自身のセルフ削除 API が 存在しない — GDPR / CCPA の消去権に相当する機能の欠落。

証拠: main/auth_manager.py の delete_user()（407 行付近）
必要な実装: 10 以上のストアを横断する削除 or 匿名化。会計上の保持義務の設計判断を伴う

3-4. ビジネスデータの永続化が存在しない

優先度: 高

マーケットプレイス・カート・ギフトカード・コミッション・モデレーション・会員ティア・ ライセンス・紹介・2FA など、事実上すべてのビジネス状態がプロセス内 Python dict のみに 存在する。PostgreSQL / Redis の基盤は存在するが、api_server.py からの 実利用は各 2 箇所のみ。プロセス再起動で全データ消失し、水平スケール不可。

証拠: grep -c "get_database_service()\|get_cache_manager()" main/api_server.py
注意: 全ストアが同一のシングルトン + Lock パターンのため機械的に移行しやすい

3-5. ログイン時に 2FA が強制されない

優先度: 中

auth_manager.login() は 2FA の有効状態を一切参照しない。永続化は機能する ようになったが、有効化してもログインが 2 要素にならない。/api/2fa/* は 独立 API として動くだけ。auth_manager.py は 193 テストを持つため 慎重な変更が必要。

証拠: grep -in "2fa\|totp\|two_factor" main/auth_manager.py → 0 件

3-6. Prometheus の実スクレイプ対象が無い

優先度: 低

/metrics は JSON レスポンス + 認証必須で、Prometheus のテキスト形式ではない。 本物のエクスポーターは独立プロセス設計で API サーバーに未統合。

証拠: docker/prometheus.yml 内のコメント参照
04

過剰 — 死んでいる / 余剰な機能

4-1. 到達不能モジュール 60/95（63%）

規模: 大

main/ 配下 95 モジュールについて、2 つの実エントリポイント （api_server.py と main.py）からの import 推移閉包を BFS で 計算した結果、60 モジュールがどちらからも到達不能。BCI・ブロックチェーン監査・NFT・ メタバース統合・エッジ AI・音声クローンなどの野心的 R&D 群がこれに含まれる。

注意: この BFS は静的 import のみを見る。subprocess 経由で起動する GUI ツールや __main__ を持つ独立 CLI は「到達不能」と出ても実際には使われうる （下の 5 節参照）。60 件の一括削除は不可、個別精査が必要。

再検証用スクリプト（リポジトリルートで実行）: 
import re, glob, os
def local_imports(path):
    src = open(path, encoding="utf-8", errors="ignore").read()
    mods = set()
    for m in re.finditer(r"from \.(\w+) import", src): mods.add(m.group(1))
    for m in re.finditer(r"^\s*import (\w+)$", src, re.MULTILINE): mods.add(m.group(1))
    for m in re.finditer(r"from (\w+) import", src): mods.add(m.group(1))
    return mods
files = {os.path.basename(p)[:-3]: p for p in glob.glob("main/*.py")}
graph = {n: local_imports(p) & set(files) for n, p in files.items()}
def reach(root):
    seen, stack = {root}, [root]
    while stack:
        for d in graph.get(stack.pop(), set()):
            if d not in seen: seen.add(d); stack.append(d)
    return seen
dead = sorted(set(files) - reach("api_server") - reach("main"))
print(len(dead), dead)
